## Visão Geral

A API NTX Pay México foi desenhada para que clientes vindos da **Voluti SPEI** (`api.spei.magenpay.io`) consigam migrar com o mínimo de mudanças no código:

- **Aliases de campo** — `payer_name`, `receiverName`, `conciliationId`, `bankCode`, etc. continuam aceitos
- **amount em string** — `"100.00"` continua válido (convertemos internamente para centavos)
- **Endpoint de get-transaction equivalente** — `GET /api/spei/transaction/{externalId}` em vez de `GET /api/v1/transaction/{conciliationId}/conciliation`
- **CLABE de 16 dígitos** (cartão de débito) aceita no cash-out
- **Auth muda** — Voluti usa HMAC `API_KEY:NONCE:SIGNATURE`; NTX Pay usa **certificado X.509 (mTLS) + OAuth 2.0 → JWT Bearer**. Essa é a maior mudança.

## 1. URL Base & Auth

| Item | Voluti | NTX Pay |
|---|---|---|
| Host produção | `api.spei.magenpay.io` | `sandbox.ntxpay.com` |
| Host sandbox | — | `sandbox.ntxpay.com` |
| Versionamento | `/api/v1` | `/api` |
| Auth header | `Authorization: API_KEY:NONCE:SIGNATURE` | `Authorization: Bearer <JWT>` |
| Obtenção da credencial | API key + secret entregues pelo painel | `POST /api/signup` → `clientId/clientSecret`; depois `POST /api/auth/token` com cert X.509 |

Para o auth, veja [Autenticação](/api-reference/guides/authentication).

## 2. Mapa de Endpoints

| Voluti | NTX Pay |
|---|---|
| `POST /api/v1/transaction/cashin` | `POST /api/spei/cash-in` |
| `POST /api/v1/transaction/cashout` | `POST /api/spei/cash-out` |
| `GET /api/v1/transaction/balance` | `GET /api/balance` |
| `GET /api/v1/transaction/{conciliationId}/conciliation` | `GET /api/spei/transaction/{externalId}` |
| `POST /webhooks/subscribe` | `POST /api/webhooks-config` |

## 3. Payload Cash-In

Você pode manter o payload **exatamente como envia para a Voluti** — aceitamos os mesmos campos como aliases:

```bash
# Voluti
curl -X POST https://api.spei.magenpay.io/api/v1/transaction/cashin \
  -H "Authorization: API_KEY:NONCE:SIGNATURE" \
  -d '{
    "amount": "100.00",
    "payer_name": "Juan Perez",
    "conciliationId": "order_abc_123",
    "generateCheckout": true
  }'

# NTX Pay — payload IDÊNTICO funciona
curl -X POST https://sandbox.ntxpay.com/api/spei/cash-in \
  -H "Authorization: Bearer $JWT" \
  -d '{
    "amount": "100.00",
    "payer_name": "Juan Perez",
    "conciliationId": "order_abc_123"
  }'
```

Internamente convertemos para o formato canônico:
- `amount: "100.00"` → `amountCentavos: 10000`
- `payer_name` → `customerName`
- `conciliationId` → `externalId`
- `generateCheckout` é ignorado — sempre retornamos `checkoutUrl` quando aplicável

Quando estiver pronto, recomendamos migrar para o formato canônico:

```json
{
  "amountCentavos": 10000,
  "customerName": "Juan Perez",
  "externalId": "order-abc-123"
}
```

A resposta sempre inclui `conciliationId` (espelhando o `externalId`) para facilitar parsing client-side durante a transição.

## 4. Payload Cash-Out

Mesma estratégia. Aliases aceitos:

| Voluti | NTX Pay canônico | Aceito como alias? |
|---|---|---|
| `amount: "10.00"` (string) | `amountCentavos: 1000` | |
| `receiverName` | `beneficiaryName` | |
| `receiverClabe` (18 ou 16) | `destinationClabe` (18 ou 16) | |
| `bankCode` | — (não usamos) |  aceito-e-ignorado |
| `conciliationId` | `externalId` | |

CLABE de **16 dígitos** (cartão de débito) é aceita no NTX Pay.

```bash
# Voluti
curl -X POST https://api.spei.magenpay.io/api/v1/transaction/cashout \
  -d '{
    "amount": "100.00",
    "receiverName": "Maria Lopez",
    "receiverClabe": "012180001234567890",
    "bankCode": "002",
    "conciliationId": "payout_001"
  }'

# NTX Pay — mesmo payload aceito
curl -X POST https://sandbox.ntxpay.com/api/spei/cash-out \
  -H "Authorization: Bearer $JWT" \
  -d '{
    "amount": "100.00",
    "receiverName": "Maria Lopez",
    "receiverClabe": "012180001234567890",
    "bankCode": "002",
    "conciliationId": "payout_001"
  }'
```

## 5. Consulta de Transação (polling)

O endpoint dedicado mantém a semântica 1-para-1 com o `conciliation` da Voluti:

```bash
# Voluti
curl -X GET https://api.spei.magenpay.io/api/v1/transaction/order_abc_123/conciliation

# NTX Pay
curl -X GET https://sandbox.ntxpay.com/api/spei/transaction/order_abc_123 \
  -H "Authorization: Bearer $JWT"
```

Retorna um único registro. A diferença é o esquema do response — veja [`GET /api/spei/transaction/{externalId}`](/api-reference/endpoints/spei-transaction).

## 6. Saldo

```bash
# Voluti
GET /api/v1/transaction/balance

# NTX Pay
GET /api/balance
```

Estrutura do response:

```json
// NTX Pay
{
  "availableCentavos": 4873490,
  "pendingCentavos": 0,
  "currency": "MXN"
}
```

Para converter para o formato que você usava com a Voluti (string com decimais), divida por 100.

## 7. Webhooks

A Voluti aceita 3 URLs separadas (`default_url`, `cashin_url`, `cashout_url`). No NTX Pay, você cria **um webhook por configuração** com a lista de eventos que ele deve receber:

```bash
# Voluti
POST /webhooks/subscribe
{
  "default_url": "https://meu-servidor.com/webhooks",
  "cashin_url": "https://meu-servidor.com/webhooks/cashin",
  "cashout_url": "https://meu-servidor.com/webhooks/cashout"
}

# NTX Pay — uma config por URL
POST /api/webhooks-config
{ "url": "https://meu-servidor.com/webhooks/cashin", "events": ["cash_in"] }

POST /api/webhooks-config
{ "url": "https://meu-servidor.com/webhooks/cashout", "events": ["cash_out", "refund_in", "refund_out"] }
```

A validação HMAC continua, mas com headers diferentes:

| Voluti | NTX Pay |
|---|---|
| Header de assinatura não documentado publicamente | `X-NTXPay-Signature: sha256=<hex>` |
| — | `X-NTXPay-Timestamp`, `X-NTXPay-Delivery`, `X-NTXPay-Event` |

Detalhes em [Implementação de Webhook](/api-reference/guides/webhooks/implementation).

## 8. Checklist de Migração

<Steps>
  <Step title="Onboarding">
    Crie a conta com `POST /api/signup` (use `isSandbox: true` para testes). Guarde `clientId` + `clientSecret`.
  </Step>
  <Step title="Receber o certificado">
    Solicite o certificado X.509 (mTLS) ao NTX Pay para o ambiente correspondente (sandbox / produção).
  </Step>
  <Step title="Trocar o auth">
    Substitua a montagem do header HMAC pelo fluxo `POST /api/auth/token` (cert + clientId/clientSecret) → guarde o JWT por ~10 minutos.
  </Step>
  <Step title="Trocar o host">
    De `api.spei.magenpay.io` → `sandbox.ntxpay.com`. Os payloads continuam iguais (aliases aceitos).
  </Step>
  <Step title="Reconfigurar webhooks">
    Crie 1 ou mais `POST /api/webhooks-config`. Ajuste seu handler para validar o HMAC NTX Pay (header `X-NTXPay-Signature`).
  </Step>
  <Step title="Migrar para o canônico (opcional)">
    Após estabilizar, troque `amount` por `amountCentavos`, `payer_name` por `customerName`, etc. Não é obrigatório — os aliases continuam aceitos.
  </Step>
</Steps>

## Limitações conhecidas

- **`generateCheckout`** — ignorado. Sempre retornamos `checkoutUrl` no response quando aplicável.
- **`bankCode`** — aceito mas ignorado. A CLABE de 18 dígitos já carrega o ISPB do banco.
- **HMAC API_KEY:NONCE:SIGNATURE** — não suportamos esse esquema. O auth é obrigatoriamente mTLS + JWT.
