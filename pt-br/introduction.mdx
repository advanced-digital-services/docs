## Arquitetura

O NTX Pay México expõe um gateway público stateless (`advanced-backend-public-ms`) que coordena com os microsserviços internos. O cliente consome **apenas** os endpoints documentados aqui.

```
Cliente → NGINX/ALB (TLS + cert) → public-ms → { account-ms, mexico-ms (smartfastpay|sandbox) }
```

## Grupos de Endpoints

| Grupo | Endpoints |
|---|---|
| **Auth** | `POST /api/auth/token` |
| **Signup** | `POST /api/signup` |
| **Balance** | `GET /api/balance` |
| **SPEI** | `POST /api/spei/cash-in`, `POST /api/spei/cash-out` |
| **OXXO** | `POST /api/oxxo/cash-in` |
| **Transactions** | `GET /api/transactions` |
| **Webhooks Config** | `GET`, `POST`, `DELETE /api/webhooks-config[/:id]` |

## Convenções

- **Prefixo**: `/api` em todos os endpoints (exceto `/health/*`, internos)
- **Valores**: centavos MXN (inteiros)
- **Datas**: ISO 8601 UTC
- **Autenticação**: `Authorization: Bearer {jwt}` (exceto `/api/auth/token` que usa certificado X.509, e `/api/signup` que é público)
- **Rate-limit global**: 60 req/min; `/api/transactions` 30 req/min por conta

## Status de Transação

Estados possíveis (variam por endpoint):

- `PENDING` — emitida, aguardando ação externa
- `CONFIRMED` — liquidada com sucesso
- `FAILED` — rejeitada
- `EXPIRED` — expirou sem completar (típico em OXXO)

## Idempotência

Use `externalId` no request — recomendado em todos os cash-in/cash-out. Em caso de erro de rede, reenvie a mesma requisição: o NTX Pay deduplica.
